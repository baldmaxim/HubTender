## Auth conflict policy

Auth schema never silently skips and is never overwritten. `ALLOW_PROD_OVERWRITE` does not apply here. Three explicit policies:

| Policy | Used by | Behavior |
|---|---|---|
| `AUTH_FAIL_BY_DEFAULT` | default for `auth.users` / `auth.identities` | INSERT without ON CONFLICT. Duplicate id / email / (provider, provider_id) → fail with masked diagnostic. |
| `AUTH_RESUME_IF_IDENTICAL_ONLY` | `--resume` for already-completed auth tables | SELECT existing PROD row; compare all non-password fields by sha256 fingerprint AND compare `sha256(encrypted_password)`. Skip only if both match; otherwise fail. |
| `AUTH_BOOTSTRAP_MISSING_IDENTITY_ONLY` | post-import bootstrap of email-identities | Creates `auth.identities` row for users without one, gated on `PROD_ENABLED_AUTH_PROVIDERS` containing `email`. |

## Auth collisions blocked by preflight (05_prepare_prod)

`05_prepare_prod` runs a read-only preflight that compares OLD export against PROD before any auth write. Findings (all with masked diagnostics) are listed in `prepare_status.json::auth_collisions`. Categories:

| kind | meaning |
|---|---|
| `auth_users_id_email_mismatch` | same `id` in OLD and PROD, but `email` differs |
| `auth_users_password_hash_differs` | same `id`, but `sha256(encrypted_password)` differs |
| `auth_users_email_collision_different_id` | same `email`, but PROD `id` ≠ OLD `id` |
| `auth_users_already_present_identical` | identical OLD row already in PROD; safe to skip under `--resume` |
| `auth_identities_user_id_mismatch` | same identity `id`, different `user_id` |
| `auth_identities_provider_mismatch` | same identity `id`, different `provider` |
| `auth_identities_pair_collision_different_user` | same `(provider, provider_id)` under a different `user_id` |

If non-`resume_safe` collisions are present → prepare fails → import refuses to run.

## Public tables

Per-table conflict policy (no silent `DO NOTHING` as default):

| Policy | When |
|---|---|
| `FAIL_BY_DEFAULT` | default — `INSERT` без `ON CONFLICT`; duplicate raises with a redacted key + remediation hint |
| `SKIP_IF_IDENTICAL` | seed tables (`roles, units, construction_scopes, tender_statuses, markup_parameters, cost_categories, detail_cost_categories`): SELECT existing → compare row fingerprint → skip if equal, FAIL if differ |

## Disabled triggers during import

Triggers that were `ALTER TABLE … DISABLE TRIGGER`-ed BEFORE the corresponding table import and re-enabled in a `finally` block. The two known triggers cause unconditional data duplication on every imported row — disabling is therefore mandatory for any non-trivial import.

| Table | Trigger | Why disable |
|---|---|---|
| `public.tenders` | `trigger_auto_create_tender_registry` | Function `auto_create_tender_registry()` does an unconditional `INSERT INTO tender_registry`. Without disable, every imported tender creates a NEW registry row (fresh UUID) regardless of whether one already exists for that `tender_number`. Required even when PROD is empty. |
| `public.boq_items` | `trg_boq_items_audit` | Function `log_boq_items_changes()` inserts an audit row per BOQ row touched. Without disable, the audit table gets inflated by `len(boq_items)` rows beyond what we import from `boq_items_audit.ndjson`. Required even when PROD is empty. |

`pg_notify` triggers (`trg_notify_row_change_*`) are NEVER disabled — they emit websocket events but there are no subscribers during the cutover window, so the noise is harmless.

`session_replication_role` is NEVER used (it requires superuser privileges that Supabase service_role lacks). System/internal triggers are NEVER touched.
