## Final status

shipping the SPA against an unconfigured SMTP backend will result in
production users never receiving emails. Until SMTP creds land on the
server (separate ops task), do NOT flip the prod `/forgot-password` UI
visible for end users; the dev/staging path works end-to-end.

## Endpoints

| Method | Path | Auth | Body | Response |
|---|---|---|---|---|
| POST | `/api/v1/auth/forgot-password` | public | `{email}` | 200 `{success: true, reset_url?: "https://…?token=…"}` — reset_url ONLY in non-prod when SMTP not configured |
| POST | `/api/v1/auth/reset-password` | public | `{token, new_password}` | 204 on success |
| POST | `/api/v1/auth/change-password` | authed (Bearer) | `{current_password, new_password}` | 204 on success |

Error mapping:

| Code / endpoint | HTTP | Trigger |
|---|---|---|
| `forgot-password` (any) | 200 | always — anti-enumeration |
| `reset-password` `ErrResetTokenNotFound/Used/Expired` | 401 (same generic detail) | invalid / replayed / expired |
| `reset-password` `ErrPasswordTooShort` | 400 | < 6 chars |
| `change-password` `ErrInvalidCredentials` | 401 | wrong current password |
| `change-password` `ErrPasswordTooShort` | 400 | < 6 chars |
| `change-password` no Bearer | 401 | middleware reject |
| `change-password` `ErrAccountMissing` | 403 | user gone after JWT issued |

## Safety highlights

- **Plaintext password** hashed via `auth.HashPassword` (bcrypt cost 10, Supabase-compatible `$2a$10$…` prefix). Never logged. Never persisted other than as hash.
- **Reset token** generated by the existing 256-bit CSPRNG that mints refresh tokens. Persisted ONLY as SHA-256 hex hash in `app_auth.password_reset_tokens`. Plaintext leaves the process only in the email body (or, in dev with no mailer, in the `reset_url` field — guarded by `APP_ENV != "production"`).
- **Anti-enumeration**: `/forgot-password` always 200 with same body shape regardless of whether the email exists; mailer-not-configured in prod is silently swallowed + loud server log warning.
- **Single-use** enforced by `used_at`; **expiry** by `expires_at`; default TTL **60 min** (configurable per-Service via `WithResetTokenTTL`).
- **Refresh-token revoke** after both `Reset` and `ChangePassword` (`RevokeAllUserRefreshTokens`) — any prior session is invalidated, user must re-login on every tab/device.
- **`change-password`** re-verifies `current_password` against the live bcrypt hash before update — a compromised access token can't change credentials silently.
- **Logging**: only event types (`password_reset_requested`, `password_reset_success`, `password_changed`, `password_change_failed`, `password_reset_requested_unknown_email`); never token/hash/password values. Emails surfaced as a `sha8` 8-char digest in info logs for correlation without leaking the address.

## Email provider abstraction

[`backend/internal/auth/email.go`](../../backend/internal/auth/email.go) — `Mailer` interface + two impls:

- **`SMTPMailer`** — minimal `net/smtp` wrapper, 10s timeout, plain text body, supports STARTTLS via the SDK default when `SMTP_PORT=587`. PLAIN auth when `SMTP_USER` set.
- **`NoopMailer`** — `IsConfigured() == false`, `Send()` returns `ErrMailerNotConfigured`. Used when `SMTP_HOST` is empty.

`NewSMTPMailer(cfg)` returns `NoopMailer` when host is empty — call sites just use `mailer.IsConfigured()` for the dev/prod branch.

### Config env (new)

```env
APP_ENV=production            # development | staging | production (default: development)
APP_BASE_URL=https://tender.su10.ru    # required when SMTP_HOST set; used to build reset URLs

# All optional; empty SMTP_HOST → NoopMailer (no mail sent)
SMTP_HOST=
SMTP_PORT=587
SMTP_USER=
SMTP_PASSWORD=
SMTP_FROM=
```

On startup the BFF logs `password-recovery flow configured mailer_configured=<bool> app_env=… app_base_url=…` so operators can verify wiring at a glance.

## Deploy recommendation

Backend + frontend both:

```bash
# After git push origin main:
bash scripts/deploy-production.sh both
```

**Before** flipping the `/forgot-password` UI to the new app-mode flow for end users, the operator MUST:

1. Add real SMTP creds to `/srv/sites/tender.su10.ru/server/.env.prod`:
   ```
   APP_ENV=production
   APP_BASE_URL=https://tender.su10.ru
   SMTP_HOST=<provider>
   SMTP_PORT=587
   SMTP_USER=<user>
   SMTP_PASSWORD=<password>
   SMTP_FROM="TenderHUB <noreply@su10.ru>"
   ```
2. Restart BFF (`bash scripts/deploy-production.sh backend`) and verify
   the startup log line: `mailer_configured=true app_env=production`.
3. Send a test reset to an operator-owned email and confirm delivery.

Until step 3 is verified, users on prod hitting `/forgot-password` will
see the generic success toast but NO email — by design (anti-enumeration)
no error surfaces to them. The operator's server log will show
`forgot-password called but mailer is not configured in production`.

### Post-deploy operator smoke (browser)

1. Open `/forgot-password`, submit an existing email.
2. Expect generic "Если email зарегистрирован..." toast.
3. Operator (or recipient) clicks the link in email → `/reset-password?token=…`.
4. Enter new password ≥ 6 chars → 204 → redirect to `/login` after 800 ms.
5. Login with new password → 200. Old password → 401.
6. All other tabs / devices: next API call → 401 (refresh tokens were revoked).
7. Sanity: submit an UNKNOWN email at `/forgot-password` → same generic toast, no email sent.
8. Sanity: replay the used reset link → 401 toast.

## Rollback note

- Frontend rollback: redeploy a prior bundle (or restore `public.backup-…`). Old Supabase-mode forgot/reset returns. Backend new endpoints stay (idle until called).
- Backend rollback: revert the commit, redeploy backend. Frontend will get 404 on `/forgot-password` POST → toast "forgot-password failed" — degraded but not data-corrupting.
- DB rollback: no schema changes; only INSERTs into `app_auth.password_reset_tokens`. Stale rows expire automatically via `expires_at`.
