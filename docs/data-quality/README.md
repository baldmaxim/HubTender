# Проверка данных тендеров — база знаний правил

Каталог правил, по которым TenderHUB ищет ошибки и нелогичности в тендерах.
Это **источник истины**: правило описано так, чтобы его мог прочитать и поправить
инженер-сметчик, а не только разработчик.

## Роли слоёв

| Слой | Где живёт | Что делает |
|---|---|---|
| **MD-правило** | [`backend/internal/quality/rules/<code>-<slug>.md`](../../backend/internal/quality/rules/) | Смысл, обоснование, подтверждённые примеры. Читают человек и LLM. |
| **SQL внутри правила** | блок ` ```sql ` в том же файле | Исполнение. Детерминированно, быстро, гоняется на каждом тендере. |
| **Вердикт инженера** | таблица `quality_acknowledgements` | Замер: «ошибка» / «принято, это норма» + причина. |
| **Разведка** | Cursor, вручную, по регламенту | Поиск **новых** классов. См. [DISCOVERY.md](DISCOVERY.md). |

> **Почему правила лежат в `backend/`, а не в `docs/`.** Их встраивает в бинарь
> `go:embed`, а он не умеет выходить за пределы своего пакета; вдобавок образ
> собирается из контекста `backend/`, куда `docs/` не попадает. Поэтому файлы правил —
> рядом с кодом, который их исполняет, а этот README остаётся точкой входа.

Нейронки в проде нет. Правила выполняет Go BFF своим пулом pgx; страница показывает
находки и собирает вердикты. LLM участвует только на этапе авторинга и разведки —
вне продакшена.

## Формат правила

Файл `rules/<code>-<slug>.md`:

```md
---
code: T
title: Сумма привязанных материалов ≠ количеству работы
severity: error | warning | info
money: yes | no
status: active | draft
---
## Суть
Что за ошибка и почему это ошибка. Этот текст показывается инженеру на странице —
пишите так, чтобы он объяснял сам себя, без разработчика.

## SQL
```sql
-- один блок; параметр $1 = tender_id
```

## Подтверждено
- TP: <тендер, позиция, факт>
- FP: <легитимные случаи, которые правило обязано пропускать>

## Замер (дата)
Сколько находок и на какую сумму на момент замера.
```

### Обязательный контракт SQL

Ровно один блок ` ```sql `, ровно один параметр `$1` = `tender_id`, и строго такой
набор колонок (порядок важен — бэкенд читает по именам, но проверяет полноту):

| Колонка | Тип | Назначение |
|---|---|---|
| `tender_id` | uuid | контроль области видимости |
| `position_number` | numeric | № п/п позиции — по нему инженер находит строку |
| `item_no` | text | номер позиции заказчика (может быть NULL) |
| `entity_id` | uuid | id строки `boq_items` или `client_positions` — ключ вердикта |
| `fingerprint` | text | md5 значимых значений; **меняются данные → вердикт слетает** |
| `detail` | text | человекочитаемое описание с числами |
| `money_delta` | numeric | денежный эффект, NULL если неприменимо |

`fingerprint` — не формальность. Вердикт «принято» привязан к нему, поэтому
подтверждённая находка всплывает заново, как только значения в строке изменились.
Включайте в отпечаток ровно те поля, изменение которых должно снимать подтверждение.

## Как добавить правило

1. Заведите `rules/<code>-<slug>.md` по формату выше. Код — свободная буква/слово,
   лишь бы не конфликтовал с существующими.
2. Прогоните SQL на проде **только на чтение** и впишите результат в «Замер».
3. Найдите хотя бы один TP и, если есть, FP — без них правило нельзя регрессить.
4. Новое непроверенное правило заводится как `status: draft` — бэкенд его не выполняет,
   пока вы не переведёте в `active`.
5. Бэкенд подхватывает файл автоматически (`go:embed`), пересборка образа обязательна.

## Дисциплина порогов

Порог — часть правила, а не деталь реализации, и он подбирается **по замеру, а не на
глаз**. Показательный пример — правило **I**: допуск ≥ 1 ₽ даёт 2 568 находок,
допуск ≥ 100 ₽ — 621 находку, **при одной и той же сумме расхождений (−106.3 млн ₽)**.
То есть 1 947 находок были шумом, за которым нет денег. Без допуска вообще правило
выдаёт 28 737 строк, из них 26 169 — арифметика с плавающей точкой.

Правило: подняли порог — **перезамерьте и покрытие, и число находок**. Если сумма не
изменилась, а находок стало меньше, порог поднят правильно. Если поехала сумма —
вернитесь назад. Если правило шумит, порог ужимается и замер переснимается.

Точность правила считается из вердиктов: `ошибки / (ошибки + принято)`. Ниже ~50% —
правило шумит: ужимайте порог, добавляйте исключения или переводите в `draft`.
